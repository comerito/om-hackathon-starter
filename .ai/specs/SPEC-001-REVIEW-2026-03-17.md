# SPEC-001 Review Report — Three-Perspective Analysis

**Spec Under Review:** `SPEC-001-2026-03-17-hackon-platform.md`
**Date:** 2026-03-17
**Perspectives:** Enterprise Architect, Senior UX Designer, Business Analyst

---

## Executive Summary

The spec is comprehensive and demonstrates strong framework knowledge. However, the review uncovered **5 CRITICAL**, **20+ HIGH**, and **30+ MEDIUM** severity findings across all three perspectives. The issues cluster into five themes:

1. **Data integrity under concurrency** — race conditions in invitations, voting, and team membership; missing database transactions; partial failure of stage transition side effects
2. **Framework compliance gaps** — missing mutation guards, openApi exports, Zod validators for JSONB configs, DI registrations, and command pattern usage
3. **Missing user-facing experiences** — no Participants Directory, no Announcements Feed page, no "What should I do now?" prompt, mentor portal virtually empty, mobile-first not addressed
4. **Business requirements gaps** — user profile fields undefined, GDPR/RODO not addressed, several notification types missing, judge project distribution not configurable
5. **Infrastructure timing** — cross-module enrichers and caching deferred to Phase 6 but needed from Phase 1

---

## Critical Findings (Must Fix Before Implementation)

### C1. Stage transition side effects can partially fail — no recovery mechanism
**Source:** Architect E1

When advancing to DEMOS, three independent subscribers fire (publish-draft-projects, generate-demo-queue, close-team-modifications). If one fails, the competition stage is already advanced but the side effects are incomplete. The system enters an inconsistent state visible to all users.

**Fix:**
- Make each subscriber truly idempotent (check if work is done before acting)
- Add an admin "Re-run side effects for stage X" endpoint
- The Event Command Center must display side effect status (pending/completed/failed)
- Consider making critical side effects (score calculation for FINISHED) synchronous within the stage advance command

### C2. Mobile-first not addressed anywhere in the spec
**Source:** UX H1

The product spec explicitly states "Mobile-first responsive design (most participants will use phones during the event)." The implementation spec never mentions responsive design, mobile layouts, touch targets, or how complex pages (score card with sliders, demo queue with drag-to-reorder) work on a 375px screen.

**Fix:** Add a mobile design section. At minimum: scoring uses tappable number buttons not sliders, drag-to-reorder has mobile alternative (up/down buttons), project editor stacks vertically, kiosk page suggests larger screen.

### C3. Participant with no team at HACKING stage is unhandled
**Source:** UX G1

When HACKING starts, team membership is locked. Teamless participants are stuck with no guidance. The `allowSoloParticipants` flag exists but the experience for solo/unmatched participants is not defined.

**Fix:** Before advancing to HACKING, the confirmation dialog must show unmatched participant count. If `allowSoloParticipants` is false, show a clear message to teamless users. If true, auto-create solo "teams."

### C4. Stage advancement has insufficient guardrails for an irreversible action
**Source:** UX D1

Stage transitions trigger irreversible side effects (locking teams, auto-publishing drafts, calculating scores). The spec mentions a "confirmation dialog" but doesn't describe what it shows.

**Fix:** The confirmation must show: (a) current/target stage, (b) bullet list of all side effects with counts ("12 draft projects will be auto-published"), (c) warnings ("3 teams below minimum size"), (d) required checkbox "I understand this cannot be undone," (e) type-the-stage-name confirmation for DEMOS/FINISHED.

### C5. Mentor portal experience is virtually empty
**Source:** UX C1

The mentor role is defined but has zero portal-specific pages in Phase 1. Mentors log in and see a dashboard with "Assigned tracks, session requests" but session requests are Phase 2. No page to browse teams or projects in their tracks.

**Fix:** Add a minimal Phase 1 mentor page (`/portal/mentor/tracks`) showing teams and projects in assigned tracks. Defer session request flow to Phase 2 but give mentors something useful now.

## High Severity Findings

### Architecture

| # | Finding | Fix |
|---|---------|-----|
| A-H1 | **Missing CRUD events** for sponsors, prizes, panels, participations — breaks real-time refresh pattern for backend DataTables | Add standard CRUD events (created/updated/deleted) for every entity with a backend DataTable |
| A-H2 | **Competition.slug unique constraint not tenant-scoped** — violates OM's multi-tenancy contract | Change to `@Unique({ properties: ['slug', 'tenantId'] })` |
| A-H3 | **Project unique constraint on teamId only**, not `['teamId', 'competitionId']` | Change to `@Unique({ properties: ['teamId', 'competitionId'] })` |
| A-H4 | **Missing indexes** on Project.status, Project.trackId, Team.status, DemoSession.status, ProjectScore.round, TeamInvitation.inviteeId, PeerVote.voterId — all hot-path query filters | Add `@Index()` to all listed columns |
| A-H5 | **No Zod validators for JSONB config blocks** (StageConfig, DemoConfig, JudgingConfig, PeerVotingConfig) — corrupted config data cascades failures | Define Zod schemas, validate on write AND defensive-parse on read |
| A-H6 | **DemoSession missing projectId FK** — avoidable N+1 when linking demo to project | Add `projectId` to DemoSession entity |
| A-H7 | **Response enrichers deferred to Phase 6** but every list view from Phase 1 needs cross-module data (team names, member counts, track names) | Build core enrichers per-phase: competition context in Phase 1, team members in Phase 2, etc. |
| A-H8 | **Score computation timing undefined** — admin needs leaderboard during DELIBERATION but spec says "after FINISHED" only | Compute scores on-demand for leaderboard API; persist `finalScore` only at FINISHED stage |
| A-H9 | **Bulk operations not backgrounded** — auto-publish 50+ projects, generate demo queue, calculate scores could time out as inline subscribers | Implement as worker jobs dispatched from subscribers, use `useOperationProgress` for UI |
| A-H10 | **No mutation guards** (validateCrudMutationGuard) mentioned for any custom write endpoint — OM hard requirement | Add blanket statement: all custom write endpoints must implement mutation guards |
| A-H11 | **openApi export not mentioned** for any API route — OM hard requirement | Add blanket statement: every API route must export `openApi` |
| A-H12 | **No database transactions** specified for multi-step operations (invitation acceptance, score submission) | Specify `withAtomicFlush({ transaction: true })` for all multi-step writes |
| A-H13 | **Race conditions in invitation flow** — concurrent invitation acceptance can put user on two teams | Rely on DB unique constraint, handle violation gracefully; use `SELECT FOR UPDATE` on team for size limit |
| A-H14 | **Score/leaderboard data leakage** — no server-side stage check on leaderboard endpoint; judges can see each other's scores | Leaderboard endpoint must check stage >= FINISHED for portal users; scores endpoint must filter `WHERE judgeId = currentUser.id` for judges |
| A-H15 | **Anonymous incident audit log de-anonymization** — audit_logs captures userId of POST request, revealing anonymous reporter identity | Bypass standard audit log for anonymous incident creation, or use system user identity |
| A-H16 | **Soft delete inconsistency** — TeamMember has no `deletedAt` (can't track membership history), JudgingCriterion has no `deletedAt` (deleting after scores exist orphans records) | Add `deletedAt` to TeamMember (or use `leftAt`), JudgingCriterion, and JudgePanel |

## Consolidated Recommendations by Priority

### Before Implementation Begins (Blockers)

1. **Fix the data model:**
   - Tenant-scope the slug unique constraint
   - Fix Project unique constraint to include competitionId
   - Add missing indexes on all hot-path filter columns
   - Add `projectId` to DemoSession
   - Add `deletedAt` to TeamMember, JudgingCriterion, JudgePanel
   - Define ParticipantProfile entity (skills, bio, social_links, organization)
   - Add `privacyPolicyAccepted` to CompetitionParticipation
   - Add `allowVoteChange` to PeerVotingConfig

2. **Define Zod validators** for all JSONB config blocks

3. **Add framework compliance blanket rules:**
   - All custom write endpoints must implement `validateCrudMutationGuard`
   - All API routes must export `openApi`
   - All multi-step writes must use `withAtomicFlush({ transaction: true })`
   - Each module with business logic must have `di.ts`
   - Each module must have `setup.ts` with `defaultCustomerRoleFeatures`

4. **Add missing CRUD events** for all entities with backend DataTables

5. **Revise implementation plan:** move enrichers to per-phase (not Phase 6), add competition context enricher to Phase 1

### Before Phase 1 Delivery

6. **Define the portal experience contract:**
   - Add Participants Directory page
   - Add Announcements Feed page
   - Add "Your Current Task" dashboard card
   - Define stage transition UX (banner/modal)
   - Define notification UI (bell, panel, read/unread)
   - Design mobile layouts for all portal pages
   - Add WCAG 2.1 AA commitment section

7. **Add stage advancement guardrails:**
   - Detailed confirmation dialog with side effect preview
   - Side effect status tracking in Event Command Center
   - Admin re-run side effects endpoint

8. **Document deployment requirements:**
   - nginx SSE proxy timeout config
   - Redis for cross-process events in production
   - Separate worker process for persistent subscribers

### Before Demo Day (Phase 4)

9. **Harden real-time features:**
   - Add `serverTime` to SSE events for clock skew correction
   - Add SSE reconnection + state reconciliation
   - Serialize vote limit enforcement (SELECT FOR UPDATE)
   - Add caching for competition config and leaderboard

10. **Secure data access:**
    - Server-side stage check on leaderboard endpoint
    - Server-side judge ID filter on scores endpoint
    - Server-side status filter on projects endpoint (portal sees PUBLISHED only)
    - Fix anonymous incident audit log de-anonymization

## Coverage Summary

| Perspective | Critical | High | Medium | Low |
|-------------|----------|------|--------|-----|
| Enterprise Architect | 2 | 16 | 14 | 5 |
| Senior UX Designer | 3 | 12 | 16 | 5 |
| Business Analyst | 0 | 2 | 15 | 8 |
| **Combined (deduplicated)** | **5** | **22** | **30** | **12** |

**Business requirements coverage:** Of the ~80 discrete requirements in the product spec, **~60 are fully covered**, **~15 are partially covered**, and **~5 are missing**. All MVP Phase 1 "Must have" items are addressed. All Phase 2 "Nice to have" items are correctly deferred.

**Verdict:** The spec is a strong foundation that correctly leverages the Open Mercato framework. The critical findings are fixable without architectural redesign. The most impactful improvements are: (1) hardening the stage transition flow, (2) defining the mobile portal experience, (3) fleshing out the user profile data model, and (4) moving enrichers and framework compliance work earlier in the plan.
